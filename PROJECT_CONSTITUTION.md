# IntentForge AI — Project Constitution

## Constitution Version

Version: 0.1.0

---

## 1. Project Identity

**Name:** IntentForge AI

**System Identity:** Autonomous Project Intelligence and Engineering Platform

**Tagline:** From Raw Idea to Verified Project.

---

## 2. Core Purpose

IntentForge AI investigates how an AI-native system can evaluate whether a project idea deserves to be built, strengthen the idea through evidence and contradiction, transform the approved direction into a research-grounded project definition, and preserve that intent throughout autonomous engineering and verification.

---

---

## 4. Primary Research Core

The primary research core is the Intent-to-Evidence Graph.

The graph must maintain traceability between:

- ideas;
- external evidence;
- collisions;
- contradictions;
- research gaps;
- research questions;
- objectives;
- requirements;
- architecture decisions;
- modules;
- tasks;
- code changes;
- tests;
- experiments;
- results;
- publication claims.

---

## 5. Primary Research Question

How can an AI-native system transform an uncertain project idea into a justified, research-grounded, correctly built, continuously verified, and reproducible project without losing the original intent, research contribution, or architectural consistency during development?

---

## 6. Initial Supported User

The initial user is:

- a student;
- a researcher;
- or a developer

who begins with a software or AI/ML project idea.

---

## 7. Initial V1 Technical Scope

The first prototype will prioritize:

- Python;
- FastAPI;
- React;
- TypeScript;
- PostgreSQL;
- controlled AI/ML and software projects.

The architecture may support additional technologies later, but universal framework support is not a V1 requirement.

---

## 8. Core System Principles

### Evidence Before Claims

The system must not declare novelty, correctness, completion, or research support without evidence.

### Contradiction Before Confirmation

The system must actively search for evidence that challenges an idea or proposed claim.

### Small Verifiable Changes

The autonomous engineering system must prefer small, testable, reversible changes over uncontrolled whole-project generation.

### Traceability

Major decisions and artifacts must be linked to their purpose and supporting evidence.

### Human Approval at Critical Gates

The system must preserve explicit approval boundaries for major project-definition and governance decisions.

### Abstention Is Valid

The system may return insufficient evidence instead of forcing a confident decision.

---

## 9. Initial Non-Goals

The V1 research prototype will not:

- replace every commercial IDE;
- support every programming language;
- support every framework;
- guarantee that every idea is novel;
- guarantee publication acceptance;
- guarantee perfect generated software;
- perform unrestricted production deployment;
- execute arbitrary untrusted code without isolation;
- make unsupported research claims.

---

## 10. Evidence-Based Definition of Done

A project feature is not complete merely because code exists.

Completion requires the applicable combination of:

- approved requirement;
- implementation;
- successful execution;
- tests;
- integration evidence;
- traceability;
- documentation;
- research evidence;
- reproducibility evidence.

---

## 11. Change Governance

Any major change affecting:

- project identity;
- primary research question;
- research contribution;
- V1 scope;
- architecture;
- evaluation strategy;
- or evidence requirements

must update this Constitution through a new version.

---

## 12. Current State

The project is in Phase 0.

No application code should be treated as official implementation until the initial research scope, architecture decisions, and evaluation contract are frozen.
